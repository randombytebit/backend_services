Non-Functional Requirement  
Role: System  
Requirement ID: NFR-01  
Title: Brute Force Protection  
Description: System must employ a challenge-response mechanism in Passkey authentication without passwords to prevent brute-force attacks, supplemented by rate limiting using express-rate-limit to minimize automated attack risks.  
Citation: Page 3, Section III.A (Passkey Security Characteristics); Page 5, Section IV.D (Results)  
Related Title: Financial Transaction Platforms, High-Security Access Control Systems  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-02  
Title: Phishing Prevention  
Description: System must bind authentication to specific domains with private keys stored on the user's device in Passkey implementation, preventing phishing even if users interact with fraudulent sites, while TOTP serves as a backup with awareness of human-element vulnerabilities.  
Citation: Page 3, Section III.A (Passkey Security Characteristics); Page 2, Section II (Literature Review on TOTP vulnerabilities)  
Related Title: Enterprise Email Systems, Social Media Login Portals  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-03  
Title: Replay Attack Prevention  
Description: System must generate a new random challenge for each authentication attempt in Passkey, combined with timestamped signatures, and update credential counters during verification to neutralize replay attacks.  
Citation: Page 3, Section III.A (Passkey Security Characteristics); Page 5, Section IV.C.5 (Passkey Login Process)  
Related Title: IoT Device Authentication, Secure API Gateways  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-04  
Title: Password Security and Hashing  
Description: System must use Bcrypt for hashing passwords with a configurable work factor (e.g., salt rounds of 10) to generate unique, non-reversible hashes resistant to brute-force and dictionary attacks, including salting to avoid common vulnerabilities.  
Citation: Page 3, Section III.D (Web App Security); Page 4, Figure 2 (Password Encryption with Salting)  
Related Title: User Credential Management in Cloud Services, Database Security for Web Applications  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-05  
Title: Token Security and Revocation  
Description: System must implement JWT for authentication with embedded user data, signed using a secret key, and include a blacklisting mechanism in MongoDB (with 1-hour expiration) to revoke tokens before expiry, preventing compromise in case of breaches.  
Citation: Page 3, Section III.D (JSON Web Tokens); Page 4, Section III.E (Token Blacklist)  
Related Title: Session Management in E-Learning Platforms, API Token Handling in Microservices  

Functional Requirement  
Role: System  
Requirement ID: FR-01  
Title: User Registration with Verification  
Description: System must handle user registration by generating and storing a time-limited verification code in Redis (30-second expiration), sending it via Nodemailer, verifying it against the temporary user table, hashing the password with Bcrypt, and persisting the user in MongoDB upon success.  
Citation: Page 4, Section IV.C.1 (User Registration)  
Related Title: Secure Onboarding in Healthcare Portals, Account Creation in Retail Apps  

Functional Requirement  
Role: FR-02  
Title: Passkey Registration and Authentication  
Description: System must generate WebAuthn options (including challenges) for Passkey registration and login, use navigator.credentials.create() and navigator.credentials.get() on the frontend for biometric or key-based auth, verify responses with @simplewebauthn/server on the backend, store credentials (ID, public key, counter) in PasskeyCredential, and issue JWT upon success.  
Citation: Page 4-5, Section IV.C.4-5 (Passkey Registration and Login Process); Page 5, Figure 3 (Passkey Login Flow)  
Related Title: Biometric Login in Mobile Banking, Device-Bound Access in Smart Home Systems  

Functional Requirement  
Role: System  
Requirement ID: FR-03  
Title: TOTP Two-Factor Authentication  
Description: System must generate unique secret keys for users during registration using the speakeasy library, support TOTP verification as a backup to Passkey for login, and handle time synchronization issues to avoid failures or exploitation.  
Citation: Page 3, Section III.B (TOTP Implementation)  
Related Title: Backup Auth in Corporate VPNs, Multi-Factor Setup in Gaming Platforms  

Functional Requirement  
Role: System  
Requirement ID: FR-04  
Title: Google OAuth Integration  
Description: System must redirect users to Google's OAuth 2.0 authorization page via Passport.js, exchange authorization codes for access tokens, retrieve or create user profiles in MongoDB, and generate JWT for successful logins.  
Citation: Page 4, Section IV.C.3 (Google Login); Page 5, Section IV.D (OAuth2 Integration)  
Related Title: Third-Party Login in Social Networks, Federated Identity in Enterprise Software
