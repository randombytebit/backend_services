### Architectural Reasoning
- **Component Identification**: The source text describes a comprehensive authentication system implementation focusing on passwordless methods (primarily Passkey via WebAuthn) with support for hybrid elements like TOTP, email verification, and OAuth. Functional capabilities include user registration, multiple login flows (email, Google OAuth, Passkey), token management, and security features like rate limiting. Non-functional aspects emphasize security (e.g., phishing/replay protection, hashing), performance (e.g., caching with Redis), usability (e.g., seamless biometric integration), and deployment constraints (e.g., Docker modularity, browser compatibility).
- **Constraint Identification**: Security is a core quality constraint across all functions, with specific metrics like brute-force resistance via challenge-response and rate limiting. Performance constraints involve efficient data handling (Redis caching, JWT expiration). Usability requires minimal user friction (e.g., passwordless flows, clear onboarding). Feasibility includes handling edge cases like device sync issues in TOTP and recovery mechanisms.
- **Functional Logic**: The primary role is the 'System' (back-end/front-end components like Node.js/Express, MongoDB) performing authentication tasks, with 'User' interactions in flows like biometric prompts. Requirements are derived directly from methodology (Section III), implementation details (Page 3-4), and experimental setup (Section IV), e.g., API endpoints for Passkey, registration logic.
- **Quality Logic**: Non-functional requirements ensure functions meet robustness criteria, e.g., Passkey's phishing resistance (Page 2, Section III.A) ties to functional login; JWT blacklisting (Page 3) constrains token security. All are verifiable in text, avoiding invention.
- **Domain Mapping**: Functional requirements apply to web apps needing multi-factor auth (e.g., banking portals). Non-functional map to high-security environments (e.g., e-commerce for rate limiting, enterprise systems for OAuth integration). Each requirement's Related Title is unique, contextual, and limited to 2-3 examples without extras, ensuring consistency with the system's passwordless, hybrid design.
