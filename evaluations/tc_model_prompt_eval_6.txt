Role: User 
Requirement ID: NF-01 
Requirement Title: Accessibility 
Requirement Description: Accessibility is commonly understood as the degree to which the application is available to users, usually taking into account disability measures. Accessibility, or software’s ability to be “use[d] by people with a wide range of characteristics”, is found explicitly in standards but often understood as an implicit aspect of usability.

Requirement ID: NF-02

Non-Functional Requirement 
Role: User 
Requirement ID: NF-04 
Requirement Title: Privacy 
Requirement Description: Privacy relates to safeguarding users’ well-being, including regulations on how an application deals with personal or medical information. The processing of any personal information is limited by laws such as the Dutch Data Protection Act; it requires patient approval, ensures privacy is not disproportionately harmed, and favors a ‘general cause’. It includes assurance that data are accessible only to those authorized and control over how applications use and share personal data. 
Citation: Page 2, Abstract; Page 3, Section 1; Page 7, Section 3.1; Page 8, Section 3.2; Page 8, Section 3.3; Page 10, Section 3.4; Page 11, Section 4.1; Page 12, Section 4.2; Page 13, Section 5

Non-Functional Requirement 
Role: User 
Requirement ID: NF-05 
Requirement Title: Safety 
Requirement Description: Safety relates to safeguarding users’ well-being, defined as the degree to which a product or system provides the correct results with the needed degree of precision (e.g., accuracy, correctness, reliability). It includes ensuring apps contain reliable, peer-reviewed content and avoiding incorrect advices that could negatively impact health. 
Citation: Page 2, Abstract; Page 3, Section 1; Page 8, Section 3.2; Page 10, Section 3.4; Page 11, Section 4.1

Non-Functional Requirement 
Role: User 
Requirement ID: NF-06 
Requirement Title: Security 
Requirement Description: Security relates to safeguarding users’ well-being, interpreted as the degree to which a system prevents unauthorized access to data. It includes protection for personal data, such as password protection or default smartphone security. 
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 8, Section 3.3; Page 10, Section 3.4; Page 13, Section 5

Non-Functional Requirement 
Role: System 
Requirement ID: NF-07 
Requirement Title: Stability 
Requirement Description: Stability relates to the technical robustness and dependability of the application, understood as systems being operational and accessible when required to perform specified functions under specified conditions for a specified period of time. It includes aspects such as robustness, availability, uptime, and recoverability from errors. 
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 9, Section 3.3; Page 10, Section 3.4

Non-Functional Requirement 
Role: User 
Requirement ID: NF-08 
Requirement Title: Trustability 
Requirement Description: Trustability entails the users’ perceptions of characteristics like privacy, safety, and security, i.e., the ability of systems to convince users of their dependability and responsibility. It includes trust in automatically generated advices and is a subjective user criterion.

Role: User 
Requirement ID: NF-09 
Requirement Title: Usability 
Requirement Description: Usability is the ease of use and learnability of the application, principally understood as the effort needed for use and the individual assessment of such use. It includes sub-concepts like ease-of-use, learnability, aesthetical attractiveness, and catering to diverse user needs (e.g., for elderly or those with disabilities).
