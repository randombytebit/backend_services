Title request: Medical App  
Non-Functional Requirement  
Role: User  
Requirement ID: NF-01  
Requirement Title: Accessibility  
Requirement Description: Accessibility is commonly understood as the degree to which the application is available to users, usually taking into account disability measures. Accessibility, or software’s ability to be “use[d] by people with a wide range of characteristics”, is found explicitly in standards but often understood as an implicit aspect of usability.  
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-02  
Requirement Title: Certifiability  
Requirement Description: Certifiability is the necessity for a medical device or aid to be CE-certified to be used legally, based on a European Union directive that defines software assisting patients in monitoring or preventing diseases and medications as a medical device. Certifiability is the degree to which software’s behavior is approved by authorities, and is not found in any form in referenced frameworks.  
Citation: Page 2, Abstract; Page 7, Section 3.1; Page 8, Section 3.3; Page 10, Table 1; Page 11, Section 4.1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-03  
Requirement Title: Portability  
Requirement Description: Portability refers to the number of operating systems and devices the application supports. Portability is defined as “the ability of software to be transferred from one environment to another” and is present in most theories, although sometimes referred to as adaptability, installability or compatibility.  
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-04  
Requirement Title: Privacy  
Requirement Description: Privacy revolves around regulations and how an application that deals with personal or medical information should incorporate those, including limitations by the Dutch Data Protection Act on processing personal information, requiring patient approval and ensuring privacy is not disproportionately harmed. Privacy is the assurance that “data are accessible only to those authorized to have access”.  
Citation: Page 2, Abstract; Page 3, Introduction; Page 7, Section 3.1; Page 8, Section 3.2; Page 10, Table 1; Page 11, Section 4.1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-05  
Requirement Title: Safety  
Requirement Description: Safety relates to safeguarding users’ well-being when using the application. Safety is the “degree to which a product or system provides the correct results with the needed degree of precision” and is usually implied through combinations of broader concepts such as accuracy, correctness, or reliability.  
Citation: Page 2, Abstract; Page 3, Introduction; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1; Page 11, Section 4.1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-06  
Requirement Title: Security  
Requirement Description: Security relates to safeguarding users’ well-being when using the application. Security is the degree “to which a system prevents unauthorized access to data”.  
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-07  
Requirement Title: Stability  
Requirement Description: Stability relates to the technical robustness and dependability of the application. Stability is understood as systems being “operational and accessible when required” to “perform specified functions under specified conditions for a specified period of time”, including aspects such as robustness, availability or uptime, and recoverability from errors.  
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-08  
Requirement Title: Trustability  
Requirement Description: Trustability entails the users’ perceptions of characteristics such as privacy, safety, and security, including the ability of systems to convince users of their dependability and responsibility. Trust is a subjective user criterion.  
Citation: Page 2, Abstract; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1; Page 11, Section 4.1  

Non-Functional Requirement  
Role: User  
Requirement ID: NF-09  
Requirement Title: Usability  
Requirement Description: Usability is the application's ease of use and learnability. Usability is principally understood as “the effort needed for use, and on the individual assessment of such use”, and may include sub-concepts such as ease-of-use, learnability, aesthetical attractiveness, and, in some cases, accessibility.  
Citation: Page 2, Abstract; Page 3, Introduction; Page 8, Section 3.2; Page 9, Section 3.4; Page 10, Table 1
