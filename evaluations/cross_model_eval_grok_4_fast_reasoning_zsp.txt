Title request: medical app

Non-Functional Requirement 
Role: System 
Requirement ID: NF-01 
Requirement Title: Accessibility
Requirement Description: Accessibility is commonly understood as the degree to which the application is available to users, usually taking into account disability measures.
Citation: Page 8, Section 3.2 Stakeholder Interviews

Non-Functional Requirement 
Role: System 
Requirement ID: NF-02 
Requirement Title: Certifiability
Requirement Description: The necessity of certifiability, based on a European Union directive which defines that a medical device or aid has to be CE-certified to be used legally, and proposes a definition for what a medical device is; software that assists patients in monitoring or preventing diseases and medications is classed as a medical device.
Citation: Page 7, Section 3.1 Document Study

Non-Functional Requirement 
Role: System 
Requirement ID: NF-03 
Requirement Title: Portability
Requirement Description: Portability refers to the number of operating systems and devices the application supports.
Citation: Page 8, Section 3.2 Stakeholder Interviews

Non-Functional Requirement 
Role: System 
Requirement ID: NF-04 
Requirement Title: Privacy
Requirement Description: Privacy revolves around privacy regulations and how an application that deals with personal or medical information should incorporate those; the processing of any personal information is limited by the Dutch Data Protection Act; both the transfer and processing of information by third parties, entered by patients, will need to be approved by those patients; apart from getting approval, other conditions that have to be met include the aspect that the patients’ privacy is not disproportionately harmed, and that the process favors a ‘general cause’.
Citation: Page 7, Section 3.1 Document Study; Page 8, Section 3.2 Stakeholder Interviews

Non-Functional Requirement 
Role: System 
Requirement ID: NF-05 
Requirement Title: Safety
Requirement Description: Safety relates to safeguarding users’ well-being when using the application; the degree to which a product or system provides the correct results with the needed degree of precision.
Citation: Page 8, Section 3.2 Stakeholder Interviews; Page 9, Section 3.4 Recapitulation

Non-Functional Requirement 
Role: System 
Requirement ID: NF-06 
Requirement Title: Security
Requirement Description: Security relates to safeguarding users’ well-being when using the application; the degree to which a system prevents unauthorized access to data.
Citation: Page 8, Section 3.2 Stakeholder Interviews; Page 9, Section 3.4 Recapitulation

Non-Functional Requirement 
Role: System 
Requirement ID: NF-07 
Requirement Title: Stability
Requirement Description: Stability refers to the technical robustness and dependability of the application; systems being operational and accessible when required to perform specified functions under specified conditions for a specified period of time, including aspects such as systems’ robustness, availability or uptime, and recoverability from errors.
Citation: Page 8, Section 3.2 Stakeholder Interviews; Page 9, Section 3.4 Recapitulation

Non-Functional Requirement 
Role: System 
Requirement ID: NF-08 
Requirement Title: Trustability
Requirement Description: Trustability entails the users’ perceptions of privacy, safety, and security characteristics; the ability of systems to convince users of their dependability and responsibility.
Citation: Page 8, Section 3.2 Stakeholder Interviews; Page 9, Section 3.4 Recapitulation

Non-Functional Requirement 
Role: System 
Requirement ID: NF-09 
Requirement Title: Usability
Requirement Description: Usability is the ease of use and learnability of the application; the effort needed for use, and on the individual assessment of such use; sub concepts may include ease-of-use, learnability, aesthetical attractiveness, and, in some cases, accessibility.
Citation: Page 8, Section 3.2 Stakeholder Interviews; Page 9, Section 3.4 Recapitulation
