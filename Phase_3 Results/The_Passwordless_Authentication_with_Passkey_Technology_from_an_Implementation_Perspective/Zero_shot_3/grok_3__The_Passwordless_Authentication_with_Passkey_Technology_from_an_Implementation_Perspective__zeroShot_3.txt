REQUIREMENT REPORT

**Functional Requirements**

- **Role**: System Developer
- **Requirement ID**: FR-01
- **Requirement Title**: Passkey Authentication Implementation
- **Requirement Description**: The system must implement Passkey authentication using the WebAuthn standard to provide a passwordless authentication experience, leveraging asymmetric cryptography and biometric verification for enhanced security.
- **Citation**: Section III.C, Page 2-3
- **Requirement Related Title**: Passkey Implementation

- **Role**: System Developer
- **Requirement ID**: FR-02
- **Requirement Title**: TOTP as Backup Authentication
- **Requirement Description**: The system must support Time-Based One-Time Password (TOTP) as a backup or supplementary authentication mechanism during registration and login, using libraries like speakeasy for generating and verifying unique secret keys.
- **Citation**: Section III.B, Page 2
- **Requirement Related Title**: TOTP Implementation

- **Role**: System Developer
- **Requirement ID**: FR-03
- **Requirement Title**: Password Hashing with Bcrypt
- **Requirement Description**: The system must use Bcrypt for password hashing and verification, incorporating a salt with a configurable work factor (e.g., 10 salt rounds) to secure stored passwords against brute-force attacks.
- **Citation**: Section III.D, Page 3
- **Requirement Related Title**: Web App Security

- **Role**: System Developer
- **Requirement ID**: FR-04
- **Requirement Title**: JWT-Based Authentication
- **Requirement Description**: The system must implement JSON Web Tokens (JWT) for authentication and access control, embedding user access information into tokens signed with a secret key, using the jsonwebtoken library.
- **Citation**: Section III.D, Page 3
- **Requirement Related Title**: Web App Security

- **Role**: System Developer
- **Requirement ID**: FR-05
- **Requirement Title**: Token Blacklisting Mechanism
- **Requirement Description**: The system must include a token blacklisting mechanism using a MongoDB collection (TokenBlacklist) to revoke JWTs before expiration, with automatic deletion of revoked tokens after 1 hour.
- **Citation**: Section III.D, Page 3
- **Requirement Related Title**: Web App Security

- **Role**: System Developer
- **Requirement ID**: FR-06
- **Requirement Title**: Two-Factor Authentication (2FA) via Email
- **Requirement Description**: The system must implement 2FA using email-based verification codes delivered via Nodemailer, requiring users to enter a unique code during login to complete authentication.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - Two-Factor Authentication

- **Role**: System Developer
- **Requirement ID**: FR-07
- **Requirement Title**: OAuth 2.0 Integration for Google Login
- **Requirement Description**: The system must integrate OAuth 2.0 authentication using Passport.js to enable Google login, handling token exchange and user data retrieval while ensuring data protection through permission scopes.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - OAuth2 Integration

- **Role**: System Developer
- **Requirement ID**: FR-08
- **Requirement Title**: Rate Limiting for Brute-Force Prevention
- **Requirement Description**: The system must implement rate limiting using express-rate-limit to prevent brute-force attacks by limiting the number of requests from a single source within a specified time frame.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - Rate Limiting

- **Role**: System Developer
- **Requirement ID**: FR-09
- **Requirement Title**: HTTP Header Security with Helmet
- **Requirement Description**: The system must use the helmet middleware to secure HTTP headers, reducing vulnerabilities to common web attacks by enforcing secure header configurations.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - HTTP Header Security

- **Role**: System Developer
- **Requirement ID**: FR-10
- **Requirement Title**: Passkey Registration Workflow
- **Requirement Description**: The system must support a Passkey registration workflow, generating WebAuthn registration options, creating session records with challenges, and verifying credentials using biometric authentication or security keys.
- **Citation**: Section IV.C.4, Page 4-5
- **Requirement Related Title**: Logic Implementation - Passkey Registration

- **Role**: System Developer
- **Requirement ID**: FR-11
- **Requirement Title**: Passkey Login Workflow
- **Requirement Description**: The system must support a Passkey login workflow, generating WebAuthn authentication options, verifying responses, updating credential counters to prevent replay attacks, and issuing JWT tokens upon successful authentication.
- **Citation**: Section IV.C.5, Page 5
- **Requirement Related Title**: Logic Implementation - Passkey Login Process

- **Role**: System Developer
- **Requirement ID**: FR-12
- **Requirement Title**: User Registration with Email Verification
- **Requirement Description**: The system must facilitate user registration by generating and sending verification codes via email using Nodemailer, storing codes in Redis with a 30-second expiration, and hashing passwords with Bcrypt upon successful verification.
- **Citation**: Section IV.C.1, Page 4
- **Requirement Related Title**: Logic Implementation - User Registration

- **Role**: System Developer
- **Requirement ID**: FR-13
- **Requirement Title**: Session Management
- **Requirement Description**: The system must handle user sessions securely using express-session to ensure efficient and secure session management during authentication processes.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - Session Management

- **Role**: System Developer
- **Requirement ID**: FR-14
- **Requirement Title**: Caching with Redis
- **Requirement Description**: The system must use Redis for caching temporary data such as verification codes to reduce database load and improve authentication performance.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - Caching

**Non-Functional Requirements**

- **Role**: Security Architect
- **Requirement ID**: NF-01
- **Requirement Title**: Phishing Resistance with Passkey
- **Requirement Description**: The system must ensure phishing resistance by binding Passkey authentication to specific domains and storing private keys on user devices, preventing unauthorized access even on fraudulent sites.
- **Citation**: Section III.A, Page 2
- **Requirement Related Title**: Passkey Security Characteristics - Phishing Prevention

- **Role**: Security Architect
- **Requirement ID**: NF-02
- **Requirement Title**: Brute-Force Protection with Passkey
- **Requirement Description**: The system must provide robust brute-force protection through Passkey’s unique challenge-response mechanism, eliminating the use of passwords that can be guessed or cracked.
- **Citation**: Section III.A, Page 2
- **Requirement Related Title**: Passkey Security Characteristics - Brute Force Protection

- **Role**: Security Architect
- **Requirement ID**: NF-03
- **Requirement Title**: Replay Attack Prevention with Passkey
- **Requirement Description**: The system must prevent replay attacks by generating a new random challenge for each Passkey authentication attempt, combined with timestamped signatures to ensure request uniqueness.
- **Citation**: Section III.A, Page 2
- **Requirement Related Title**: Passkey Security Characteristics - Replay Attack Prevention

- **Role**: Security Architect
- **Requirement ID**: NF-04
- **Requirement Title**: System Resilience through Modular Architecture
- **Requirement Description**: The system must maintain resilience against security breaches by employing a modular architecture with Docker isolation capabilities, reducing the impact of compromises in individual components.
- **Citation**: Section IV.A, Page 4
- **Requirement Related Title**: Architectural Design

- **Role**: Security Architect
- **Requirement ID**: NF-05
- **Requirement Title**: Data Privacy in OAuth 2.0 Integration
- **Requirement Description**: The system must ensure data privacy during OAuth 2.0 authentication by implementing proper token handling and permission scopes to protect user data accessed through third-party services like Google.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - OAuth2 Integration

- **Role**: Security Architect
- **Requirement ID**: NF-06
- **Requirement Title**: Robustness Against TOTP Vulnerabilities
- **Requirement Description**: The system must mitigate TOTP vulnerabilities such as time synchronization issues and brute-force susceptibility by prioritizing Passkey as the primary authentication method and using TOTP only as a backup.
- **Citation**: Section III.B, Page 2
- **Requirement Related Title**: TOTP Implementation

- **Role**: Security Architect
- **Requirement ID**: NF-07
- **Requirement Title**: Secure Configuration Management
- **Requirement Description**: The system must securely manage configuration parameters using environment variables via dotenv to prevent exposure of sensitive information and ensure integrity of security settings.
- **Citation**: Section IV.D, Page 5
- **Requirement Related Title**: Results - Environment Variables

- **Role**: Security Architect
- **Requirement ID**: NF-08
- **Requirement Title**: User Experience and Recovery Mechanism Robustness
- **Requirement Description**: The system must ensure a robust recovery mechanism and clear onboarding process for Passkey authentication to maintain user trust and adaptability, addressing limitations in standardized recovery processes.
- **Citation**: Section IV.E, Page 5
- **Requirement Related Title**: Implementation Considerations

- **Role**: Security Architect
- **Requirement ID**: NF-09
- **Requirement Title**: Compatibility and Fallback Support
- **Requirement Description**: The system must ensure compatibility with various platforms and browsers for Passkey authentication, providing fallback methods like TOTP for unsupported environments or legacy devices to maintain accessibility without compromising security.
- **Citation**: Section IV.E, Page 5
- **Requirement Related Title**: Implementation Considerations

- **Role**: Security Architect
- **Requirement ID**: NF-10
- **Requirement Title**: Integrity of Authentication System Components
- **Requirement Description**: The system must maintain the integrity of authentication components by carefully managing interdependencies and fine-tuning configurations to avoid overly restrictive or insufficient security settings that could impact effectiveness.
- **Citation**: Section IV.F, Page 5-6
- **Requirement Related Title**: Lessons Learned
