### Architectural Reasoning for NFR-01 (Authorized Use of Sensors or Portals)
- Component Identification: Table I explicitly defines Privacy Pattern 1 as "Authorized Use of Sensors or Portals," focusing on controlling access to device hardware like sensors (e.g., camera, microphone) and portals (e.g., Bluetooth) to prevent unauthorized use, which is a core privacy constraint in mobile OS.
- Constraint Identification: The solution requires the system to enforce explicit user permission requests and provide management settings, addressing forces like accidental denials while ensuring strict control.
- Quality Logic (NFR-01): This is a non-functional requirement for the 'System' (mobile OS) to implement permission-based access, aligning with privacy assurance in RE activities for all stakeholders (engineers, developers, providers, users).
- Domain Mapping: Applicable to scenarios like app-based surveillance prevention in smartphones; related titles include mobile app permission managers in Android/iOS and IoT device access controls.
- Proof of Existence: Directly from Table I, Section IV, with example of microphone access permission.

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-01  
Title: Authorized Use of Sensors or Portals  
Description: The system shall demand the application to explicitly ask for permission from the user to access the sensors and portals. The system shall also provide a host of settings to manage these permissions.  
Citation: Xuan et al. (2014). Privacy Requirements Patterns for Mobile Operating Systems. Table I, Section IV.  
Related Title: Smartphone Hardware Access Control, Mobile App Sensor Permission System, IoT Device Privacy Gatekeeper

- Constraint Identification: The solution involves user warnings for access attempts, permission grants only upon approval, and options to withdraw/regrant, mitigating risks like accidental denials while ensuring strict control.

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-03  
Title: Guard for Personal Mobile Data  
Description: The system shall provide warnings to the user when an application intends to access user files, data and credentials. If and only if the user has granted the permission, the application can touch the corresponding resources. The user can withdraw or regrant any permissions in somewhere of the system later.

### Architectural Reasoning for NFR-04 (Privacy Protection over Mobile Cloud Services)
- Component Identification: Table IV details Privacy Pattern 4 as "Privacy Protection over Mobile Cloud Services," addressing data handling in cloud backups due to limited device storage and frequent updates.
- Constraint Identification: The solution requires data encryption before upload, secure token authentication, user notifications about backup content, and options to stop/delete data, trading off usability with privacy and recoverability.
- Quality Logic (NFR-04): A non-functional requirement for the 'System' to ensure confidential cloud interactions, involving stakeholders (engineers, developers, providers, users) in all RE activities.
- Domain Mapping: Critical for cloud-synced mobile ecosystems; related titles encompass auto-backup privacy enhancers and secure cloud file managers.
- Proof of Existence: Directly from Table IV, Section IV, with example of 256-bit AES encryption for data transmission.

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-04  
Title: Privacy Protection over Mobile Cloud Services  
Description: The system shall ensure confidentiality of data backup. The data encryption must be performed before the data sending over the Internet. Secure tokens shall be used for authentication. The user should be aware what data will be backed up to the cloud. The user also can stop the backup function, and delete unwanted data from cloud with her device any time.  
Citation: Xuan et al. (2014). Privacy Requirements Patterns for Mobile Operating Systems. Table IV, Section IV.  
Related Title: Mobile Cloud Backup Encryptor, Secure Sync Privacy Manager, Data Upload Consent System
