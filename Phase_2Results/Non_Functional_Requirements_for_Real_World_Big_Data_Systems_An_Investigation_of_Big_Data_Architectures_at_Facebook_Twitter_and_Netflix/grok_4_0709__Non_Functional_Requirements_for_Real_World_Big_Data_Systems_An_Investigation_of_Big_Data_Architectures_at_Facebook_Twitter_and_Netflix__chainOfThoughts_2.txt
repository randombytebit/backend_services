Title: Batch Data  
Description: This requirement refers to the capability to process data which is finite and usually large in volume, e.g. data archived in distributed file systems or databases. Requirement ID: FR-02  
Title: Stream Data  
Description: This requirement refers to the capability to process data which is potentially infinite and usually flowing at high velocity, e.g. monitoring data captured and processed in real-time, or close to real-time. Stream processing at Facebook is done by a suite of in-house developed applications: Puma, Swift and Stylus. Storm, one of the most popular stream processing frameworks in use today, was developed by Twitter. Netflix also uses Mesos to manage its large cluster of cloud resources, with scheduling done by Fenzo and stream processing by Mantis.[cite: Vergilio 2019]

Functional Requirement  
Role: System  
Requirement ID: FR-03  
Title: Late and Out of Order Data  
Description: This requirement relates to stream processing and refers to the capability to process data which arrives late or in a different order from that in which it was emitted. Streaming data from mobile users, for example, could be delayed if the user loses reception. All three streaming architectures utilise the concept of windows of data to transform infinite streaming data into finite windows that can be processed individually. Facebook’s Stylus utilises low watermarks, Twitter's Heron has a withLag() method, and Netflix's strategy is not detailed but implied through proprietary code.[cite: Vergilio 2019]  
Citation: Section 3.3 of Vergilio, T and Ramachandran, M (2019) Non-functional requirements for real world big data systems: an investigation of big data architectures at Facebook, Twitter and Netflix. Non-Functional Requirement  
Role: System  
Requirement ID: NFR-01  
Title: Processing Guarantees  
Description: This requirement refers to a stream system’s capability to offer processing guarantees, i.e. exactly once, at least once and at most once. While exactly once processing is ideal, it comes at a cost which could translate into increased latency. At Facebook, Stylus provides at least once semantics via Swift and Scribe. At Twitter, Heron offers effectively once semantics. Netflix uses Kafka for exactly once through idempotency and atomic transactions.[cite: Vergilio 2019]  
Citation: Section 3.4 of Vergilio, T and Ramachandran, M (2019) Non-functional requirements for real world big data systems: an investigation of big data architectures at Facebook, Twitter and Netflix. Proceedings of the 13th International Conference on Software Technologies, pp. 833-840.[cite: Vergilio 2019]  
Related Title: Reliable Messaging System, E-commerce Order Processing System, Banking Transaction Guarantee System[cite: Vergilio 2019]

Title: Distribution and Scalability  
Description: This requirement refers to the capability to distribute data processing amongst different machines, located in different data centres, in a multi-clustered architecture. Twitter's Heron uses Dhalion for dynamic scaling. Netflix's cloud-based system uses Fenzo for elastic scaling.[cite: Vergilio 2019]

Requirement ID: NFR-05  
Title: Fault Tolerance  
Description: This requirement refers to the capability of a system to continue to operate should one or more nodes fail. Ideally, the system should recover gracefully, with minimal repercussions on the user experience. Facebook implements via node independence and Scribe. Twitter addresses at multiple levels (architectural, container, JVM). Netflix uses multi-region AWS deployments for redundancy.[cite: Vergilio 2019]

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-06  
Title: Flow Control  
Description: This requirement refers to the capability to handle scenarios where the data source is emitting records faster than the system can consume. Architectures typically provide strategies for dealing with backpressure, e.g. dropping records, sampling, applying source backpressure. Facebook uses Scribe for resilience. Twitter's Heron tightens upstream components. Netflix uses ReactiveX libraries like RxJava for strategies such as buffering or sampling.[cite: Vergilio 2019]  
Citation: Section 3.9 of Vergilio, T and Ramachandran, M (2019) Non-functional requirements for real world big data systems: an investigation of big data architectures at Facebook, Twitter and Netflix. Related Title: Backpressure Management System, High-Velocity Data Ingestion System, Reactive Streaming Pipeline[cite: Vergilio 2019]
