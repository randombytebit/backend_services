Non-Functional Requirement  
Role: System  
Requirement ID: NFR-01  
Title: Batch Data Processing  
Description: The system must support the processing of finite, large-volume data stored in distributed file systems or databases, such as archived data reaching hundreds of petabytes, with daily processing volumes in the tens of petabytes. This includes using combinations of independent but communicating systems like Operational Data Store (ODS), Scuba, Hive, Laser, Hadoop clusters, Scalding, Presto, Genie, or Amazon S3 for efficient management and querying of stored data.  
Citation: Section 3.1 Batch Data (Page 3)  
Related Title: Social Media Analytics Platforms, E-commerce Data Warehousing Systems, Large-Scale Log Analysis Tools  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-02  
Title: Stream Data Processing  
Description: The system must handle potentially infinite, high-velocity data flows in real-time or near real-time, such as monitoring data captured and processed continuously. This involves using specialized applications or frameworks like Puma, Swift, Stylus, Storm, Heron, Mesos, Aurora, Fenzo, or Mantis to manage stream processing, including checkpointing and forming complex directed acyclic graphs (DAGs) for stateful or stateless processing units.  
Citation: Section 3.2 Stream Data (Page 3-4)  
Related Title: Real-Time User Engagement Tracking Systems, IoT Sensor Data Pipelines, Live Video Streaming Analytics  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-03  
Title: Late and Out-of-Order Data Handling  
Description: The system must process streaming data that arrives late or out of order, such as data delayed due to network issues like lost mobile reception. This requires mechanisms like data windows to transform infinite streams into finite processable units, low watermarks for handling delays, or methods to specify maximum lag times (e.g., via withLag() in APIs) to ensure correct ordering and completeness.  
Citation: Section 3.3 Late and Out of Order Data (Page 4)  
Related Title: Mobile App Event Logging Systems, Financial Transaction Processing Platforms, Network Traffic Monitoring Tools  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-04  
Title: Processing Guarantees  
Description: The system must provide processing guarantees for streams, including exactly-once, at-least-once, or at-most-once semantics, balancing completeness, duplication avoidance, and latency costs. This involves checkpointing for state persistence, idempotency, atomic transactions, or effective-once semantics (where data may be processed multiple times but delivered once) to handle failures without data loss or redundancy.  
Citation: Section 3.4 Processing Guarantees (Page 4)  
Related Title: Distributed Messaging Queues, E-commerce Order Processing Systems, Real-Time Fraud Detection Engines  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-05  
Title: Integration and Extensibility  
Description: The system must integrate with existing services and components while allowing future extensions to incorporate new components. This includes using messaging systems like Scribe for seamless data exchange between systems, modular architectures for decoupling internal components, or frameworks that enable flexible adaptations to different infrastructures and use cases, facilitating complex pipelines and broader adoption.  
Citation: Section 3.5 Integration and Extensibility (Page 4-5)  
Related Title: Enterprise Service Bus (ESB) Architectures, Modular Microservices Platforms, API Gateway Systems  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-06  
Title: Distribution and Scalability  
Description: The system must distribute data processing across multiple machines and data centers in a multi-clustered architecture, supporting dynamic scaling by adding or removing nodes without downtime. This involves increasing parallelism through additional brokers or buckets, using frameworks like Dhalion for automatic scaling decisions, or elastic cloud-based scaling with tools like Fenzo to manage resources efficiently.  
Citation: Section 3.6 Distribution and Scalability (Page 5)  
Related Title: Cloud-Native Distributed Computing Clusters, High-Traffic Web Application Backends, Scalable Database Sharding Systems  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-07  
Title: Cloud Support and Elasticity  
Description: The system must support deployment in the cloud to leverage economies of scale, including elasticity for scaling up or down based on demand, minimizing wastage by paying only for used resources. This includes shifting to container-based approaches (e.g., using AWS, Mesos, or Kubernetes) to decouple frameworks from infrastructure, enabling multi-cloud setups and reducing vendor lock-in for powerful, cost-effective scalability.  
Citation: Section 3.7 Cloud Support and Elasticity (Page 5)  
Related Title: Multi-Cloud Data Processing Frameworks, Elastic Compute Services for AI Workloads, Container Orchestration Platforms  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-08  
Title: Fault Tolerance  
Description: The system must continue operating despite node failures, recovering gracefully with minimal impact on user experience. This requires node independence, persistent messaging for communication, modular distributed architectures, decentralized topology management, resource isolation at JVM or container levels, and redundancy across multiple cloud regions (e.g., AWS US-East-1, US-West-2, EU-West-1) to isolate failures and enhance resilience.  
Citation: Section 3.8 Fault Tolerance (Page 5-6)  
Related Title: High-Availability Cluster Management Systems, Disaster Recovery Platforms, Resilient Microservices Architectures  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-09  
Title: Flow Control  
Description: The system must handle scenarios where data sources emit records faster than consumption rates, implementing backpressure strategies like dropping records, sampling, applying source backpressure, buffering, debouncing, or windowing. This can be achieved through persistent messaging systems for node independence, tightening upstream components to slow data flow, or using reactive libraries (e.g., RxJava) for programmatic control of emission rates.  
Citation: Section 3.9 Flow Control (Page 6)  
Related Title: Real-Time Data Ingestion Pipelines, Overload Protection in Streaming Services, Reactive Programming Frameworks  

Non-Functional Requirement  
Role: System  
Requirement ID: NFR-10  
Title: Flexibility and Technology Agnosticism  
Description: The system must allow the use of different technologies in place of existing components, promoting modularity and interoperability. This includes designing architectures that are not tightly coupled to specific tools (e.g., replacing Mesos or Aurora), supporting reactive standards for library interoperability, using containers for deployment abstraction, and enabling open-source contributions to enhance adaptability across various infrastructures and programming models.  
Citation: Section 3.10 Flexibility and Technology Agnosticism (Page 6)  
Related Title: Technology-Agnostic Middleware Platforms, Cross-Platform Data Processing Engines, Open-Source Modular Frameworks
